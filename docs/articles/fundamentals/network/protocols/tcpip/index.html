TCP/IP Stack

general

article

network , protocol , osi

This article is an overview of TCP/IP protocol stack. I will cover in detail what happens when a request to, say, google.com is sent in the browser.

There is an OSI model. It’s an empermal model. It’s not a real thing, it’s more like a blueprint for making real things. This model basically describes, what an empermal packet sent over network should approximately look like. It’s called a packed because it looks very much like one: your message wrapped in an envelope, wrapped in some paper, sent via post office and deliver to the door. Comparing this process with mail services is not uncommon, but I will use another example. It’s unreal, but it better describes the process.

MAC - person’s full name (because it’s almoust unique to device)

IP + TCP/UDP - person’s street and house

Application data - message itself.

There is this far far away world. It this world there are lots of cities, but we are going to talk about very peculiar one. It is so peculiar because of the way they are messaging. They have something that looks very much like a Russian matryoshka doll.

As you may already know, it’s a hollow doll, which has another doll inside, also hollow and which also … well, you get the idea. That’s why they are called - nested dolls. Now imagine that in this city people actually talk using this dolls.

They first write a message on a paper, they they fold it a couple of times and put into the smallest hollow doll. They write the lucky receiver’s name on it and put it inside another doll. They write a flat number on it and then put it into another doll. On that next doll they write and address (Street and house) of the lucky receiver. Finally, they put inside the biggest doll and give it to a postman on a bike 🚴‍♂️ .

A 🚴‍♂️ postman on a bike doesn’t do anything at all apart from delievering it to some points. He only knows all the routes of the city, but he doesn’t even know what he’s delivering. Upon bringing this matryoshka doll to the post office he is assigned to, he opens the outer matryoshka and handes the neseted ones to the postoffice clerk 👨‍💼.

The postoffice clerk’s job is far more difficult. He reads the address on the matryoshka he has and tells the 🚴‍♂️ postman on a bike which street and which house he is supposed to go to. They both then sit on the same bike and ride to the specified house. They are met by a housekeeper 👵 there who gets the next matryoshka from the postoffice clerk’s 👨‍💼 one and reads the name of the person to get this to. If the person is known to live in this house, she accepts the matryoshka and delivers it to the lucky receiver. This lucky receiver now opens the last matryoshka and reads his message.

A very peculiar algorithm but it works… That’s approximately how this model works. The 🚴‍♂️ postman on a bike is a physical layer (cables and radio waves) whose job is not get everyone to the correct places. The postoffice clerk 👨‍💼 is the network layer and transport layer combined. He reads the address of the house to ride to. The housekeeper 👵 is the data link layer, whose job is to identify the correct person and drop the parcel if no such person lives in the house she owns.

When you type bakerst221b.com in the address bar of your browser, your PC is doing a tremedous work behind the curtains. Let’s see, what this little workaholic elf is doing there.

He needs to pack your message into a set of nested envelopes (matryoshkas in the example above). What’s the message? Since we are talking about websites, the most common wrapper would be the HTTP wrapper. What’s an HTTP?

Network Architecture

There is a nice family of Jack and Mary. They like pears 🍐 so let’s mark their realm with one. The like this website and use it pretty often (grapes realm 🍇). But since these realms are quite far from each other, they can only communicate over some intermediary networks (network A ruled by Router A and network B ruled by Router B).

Let’s just asume that our PC with IP 192.168.1.70 and the website with IP 185.199.110.153 are on two different VLAN’s that are connected via three routers and let’s also assume our PC had to acquire MAC itseld. What would happen? Here is our current network architecture:

Green arrows and green names with IP addresses (🍐 realm) is Mary’s and Jack’s home network. You can see lots of devices connected. Mary wants to connect to bakerst221b.com using her laptop. Her requests will go to the Jack's and Mary's Router, then to the Router A, then Router B and get to Bakerst221b.com's Router. The latter will then forward the request to bakerst221b.com. So the route is 🍐 -> 🍊 -> 🍇. The response will go the same route but reversed (🍇 -> 🍊 -> 🍐).

Let’s see this work in more details using an example. One sunny day Mary decides to visit bakerst221b.com and check it for updates. She opens her laptop and opens a web browser. Then she types in the address bar https://bakerst221b.com and hits Enter. Her laptop is not panicking. It knows what to do. It needs to pack the following parcel:

XXX in red stand for some data that the PC 🖥️ doesn’t know at the moment:

  • bakerst221b.com's IP address
  • bakerst221b.com's port number
  • Its own port number that it will use to send the request (it was not assigned yet)
  • bakerst221b.com's MAC address

It sends two separate request to fill in the fields in question.

Step 1. Get the IP address

Using DNS protocol her PC asks the following:

💻 ​ “I have a human readable garbage (bakerst221b.com), I need a hardcore machine equivalent (IP address). Does anyone know who the heck bakerst221b.com is?”.

– Whom is it asking?

– A DNS server.

– Where is it?

– Her internet provider has it.

– Does Internet Provider’s DNS server has all the domain names?

– Of course not. It would blow up. It asks its fellow DNS buddies until the right one is found.

As it turns out, Mary’s laptop first tries doing this over IPv6 (red square) and fails. Two requests: for A and AAAA records (see DNS article) to get both IPv4 and IPv6 addresses for bakerst221b.com. For both requests itgot a Refused response - I don’t understand the whole internals of this, but guess IPv6 connection is not supported for my hosting server 💡 .

Then it tries to ask over IPv4 for the same both records (A and AAAA) and gets the addresses it needed (purple square):

Since there are no IPv6 addresses assigned, 🖥️ PC only needs the response for A record. Actually, there are 4 IPv4 addresses that bakerst221b.com uses. Ok, not that 🖥️ gets its IPs, he can finally sign his first envelope (the IP header). But there are also transport and datalink layers…. Step 2. Get the MAC address

As for Data Link layer, it’s a tricky one. The 🖥️ itself doesn’t do that dirty job. It just puts its router’s MAC address, knowing that the router will know what to do. Let see, what that little chap is doing. To determine MAC address (to sign the second envelope with MAC addresses) the router sends an ARP request. The below is ARP request

![Screenshot 2020-11-29 at 7.17.15 PM](images/Screenshot 2020-11-29 at 7.17.15 PM.png)

and response examples:

![Screenshot 2020-11-29 at 7.17.22 PM](images/Screenshot 2020-11-29 at 7.17.22 PM.png)

ARP in more detail can be read about here. The main idea is that the device is asking “Who has 185.199.110.153? Tell 192.168.1.70”. As soon as it gets the answer, it fills the data link layer header with the correct MAC address. But in our current example we wouldn’t see this request, since the job is done behind the router’s curtains. Please note, that ARP is only packed into data link (Ethernet) envelope 📩 , not subsequent wrappings into IP and TCP envelopes since we are not actually sending any data and don’t need IP and TCP envelopes for that. Our postoffice clerk 1 and postoffice clerk 2 are both having a rest 🌴 .

PC already knows the destination IP that he acquired with the help of DNS request: 185.199.110.153. It also knows its own MAC and IP (obviously). The only thing missing is the bakerst221b.com MAC address. ARP is how it gets this piece of the puzzle.

Ethernet (Data Link layer) is the only envelope it will use. Since Destination Address is "FF:FF:FF:FF:FF:FF" everyone in the same broadcast network will get the message, including

Ethernet (Mac Header): {
  Destination Address: "FF:FF:FF:FF:FF:FF",
  Source Address: "4c:8d:79:eb:3f:30",
  Type: 08-06 ARP
}
ARP Request: {
  Hardware Type: 1 Ethernet [10Mb],
	Protocol Type: 08-00 DoD IP,
	Length of Hardware Address: 6,
	Length of Protocol Address: 4,
	Operation code: 1 (Request),
	Sender's Hardware Address: "4c:8d:79:eb:3f:30",
	Sender's IP Address: "192.168.1.70",
	Target's Hardware Address: "00:00:00:00:00:00",
	Target's IP Address: "185.199.110.153"
	Frame padding: 18 bytes
}
