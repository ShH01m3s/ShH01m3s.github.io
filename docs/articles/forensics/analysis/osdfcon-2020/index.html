Remote Desktop Connection

RDP Bitmap Cache. RDP was developed by Windows. Small chunks of screenshots. The size of each is 64x64 bit. And also there is 1 huge sprite with all of them.

%USERPROFILE%\AppData\Local\Microsoft\Terminal Server Client\Cache
  • older - .bmc
  • newer - Cache####.bin where # is a number starting from 0

A Python2-based extraction tool - https://github.com/ANSSI-FR/bmc-tools with -b flag. Was with a bug (4 bytes offset mistake). https://github.com/brimorlabs/rdpieces/blob/master/modified-bmc-tools.py. Powershell https://github.com/gtworek/PSBits/blob/master/DFIR/DecodeRDPCache.ps1.

  1. Extract data from RDP Bitmap Cache
  2. Ensure output folder(s) and format correct
  3. Rearrange to make a whole screen (challenging).

The solution is to see this process as a puzzle. You look at the edges and try to find a corresponding piece that matches. All the math is performed by Imagemagick and SQL. Trial and error showed that the edge should be 5px in width or height (depending on how we are matching: top, bottom, left or right).

RDPieces.pl gives approximately 400 ordered results instead of 6400 unordered ones.

❗ Ransomware sometimes encrypts bin files + there is a registry key that turns this off. So, if any or both of these were done by a malicious party, this method becomes useless.

❓ Registry key.

Volatility Troubleshooting

There is a new version of Volatility out there and some changed made. So, need to update my notes on memory forensics. Older version doesn’t support hybernation files on Win8 and newer. The format is different.

Types:

  • raw
  • Hybernation (before Win8) Windows
  • Crashdumps Windows
  • Other file formats (vbox, vmem, vmss)

Acquisition errors

  • hard to find a good free tool
  • failing in the middle
  • smearing due to a long time (pointers pointing to the data that is not there anymore and written to disk)
  • corrupted memory samples
  • malware interference
  • Updates leading to new memory structures

You need the correct profile:

  • kdbgscan, imageinfo (slow and shows info only about the first KDBG block that it encounters) for Windows. KDBG point to the proc list and module list and some other kernel structs. This info is used to determine the profile.
  • grep for BOOT_IMAGE for Linux to find kernel info
  • mac_get_profile plugin for Mac

Now we don’t have to specify the profile. Just download all the profiles and it is going to figure it out itself. For Windows even easier.

Troubleshooting

  1. Check whether the memory sample is even valid:
memory.img tr -d '\0' | read -n 1 || echo "Empty"
xxd memory2.img | grep -v "0000 0000 0000 0000 0000 0000 0000 0000" | less -I
  1. Missing dependencies. For example, distorm3.
  2. Acquisition smearing (buggy acquisition, incomplete or changes in memory).

When everything is broken you’ll be likely see this:

Try strings.exe. Try scanning plugins. Try hex editor. Try bulk_extractor https://github.com/simsong/bulk_extractor. Try mftparser for vol2.

When something is broken, but not everything you get this:

Try psscan. If it renders any output, try using a DTB (PDB) from a different process.

❓ What is DTB (PDB)? DTB - Directory Table Base and Process Table Base. Volatility uses some default process to get the base of this table. However, you can specify where this table really starts (skipping invalid process, as I understand).

Running psscan we can determing any valid process and use its address as the new base.

python vol.py -f messed.dmp --profile=Win psscan
# note the address of any system proces
python vol.py -f messed.dmp --profile=Win --dtb=0xblahblah pslist
# if one fails, try the next one. May be you will get lucky
  1. Multiple KDBG blocks. Some are valid and some not.
python vol.py -f mem.raw kdbgscan

To determine whether the profile is the correct, note Build string, KDBGHeader, Profile Suggestion, Kernel base.

PsActiveProcessHead, PsLoadedModuleList must have some processes and modules:

KdCopyDataBlock (V) - address of KDBG block in memory. Put that on the cmd to speed up since vol.py looks for KDBG block internally each time it’s run. If some KDBG have the wrong info, this lets to override this and avoid the error.

python vol.py -f mem.raw --kdbg=0xf805784bb744 --profile=Win10x64_18362 pslist
  1. If the machine was updated and the structures are different. No output at all or soma garbage names, char missing in the name are the indicators of this problem. Create a new profile or new vtypes (🗒 learn how to do this). pslist works well since it uses kdbg to look for the processes based on the pointers. psscan uses signatures and will therefore fail.

Forged Image Detection

https://29a.ch/photo-forensics/#forensic-magnifier

Images can be forged or cloned. Regions can be manipulated with. Hard to detect with naked eye. ❗ Social Media strips-off metadata from uploaded images which makes tracking of the image difficult.

Main goals are: detect whether the image was cloned and track the source of the image. Forensically and JPEGSnoop open-source tools. They detect tampering.

  1. Magnification. Autocontrast will not disturb the colors too much. Autocontrast by channel will.
  2. Clone Detection. The most important. Minimal similariyt ~ 0.5. Minimal detail ~ 0.1. Minimal cluster size - the number of blocks that two regions need to share in order to be considered clones ~ 14.
  3. Error-level analysis. Tells that something was compressed multiple times. For example, when you paste an image on another image and save, the are several compressions in place: for the original image and for the pasted one. Use noise to determine that. Spot the artifacts that have been implanted on an image by compressing it multiple times. Because of the noise these parts can’t be compressed much.
  4. Noise analysis. Self-explanatory.
  5. JPEG analysis. Computes the Quantization table, which specify the way an image has been compressed. Each cell 0-255. To compress an image the pixel intestines are modified by the software app in the range of -128 to 127 and new quantization table is calculated. The value of standart - 95, which is auto value when the image is not processed. Adobe (Photoshop quality) uses 85. The order of different sections. SOI - image start, EOI- end of image. Multiple SOI and EOI are indicative of pasting other images. Application segments, APP0(JPEG version, screen and printing resolution), APP1 (date/time, focal length, aperture), APP13 (if processed with Adobe Photoshop). SOS - image was compressed.
  6. Structural and String Analysis. When there is no EXIF is useful. FBMD01000a9... indicated web upload (Facebook).

This one was processed with Adope, since APP13 is there. Also, JPEG images sometimes contain Comment section.

JPEGSnoop for image tracking extracts embedded info. Identifies “Original Transmission Reference” (number or an id embedded in the image, provided by the creator or image provider used for transmission and tracking purposes).

Based on the compression signature of the cloned image, it generates a list of devices/software, which could have been used for taking the image or creating it. Track the device and model (when no EXIF especially).

Limitation

Magnification blurs the colors so difficult to detect tampering. GPS might not be available with the image was taken from the device which either hasn’t or disabled GPS tracking. Social media strip-off metadata. ❗ JPEG Analysis can identify quantization tables only if image is processed and edited using Adobe.

Autopsy Update

APFS Disk images support, XRY Text File Reports. Chromium support. More Windows artifacts. New Picture Analyzer (former EXIF Module) + converts HEIC to JPEG and adds as children.

❓ what’s HEIC?

Flags files based on name and path. For example, googlesync.exe goes under category Cloud Storage. Contribute rules. Incorprorating scoring from CyberTriage.

Central Repository. Enabled by default. Doesn’t correlate by default.

Drone analyser. New, based on DHS S&T Contact (DAtCon to parse DJI drone data and creates track points for where the drone was). GPX module creates track points, way points, etc from non-drones. Can be displayed in the new Map viewer.

iOS Analyzer! Finally! Uses iLEAPP. .tar as input file. 4.18 should support disk images, more artifacts and aLEAPP.

New Context Viewer. Where did it come from (downloads, attachments)? When or where was it used (MRU keys, Prefetch, etc)?

Data Source Summary. About this data source, about user activity ❗.

Discovery. Search navigation. Items are organized in groups. See copies of the artifacts.

Maps. Geo from iOS, Android, Drone, EXIF.Uses Bing or OpenStreetMap.

Interesting modules:

  1. W10-FaceMesanger@Autopsy is Data Source Module. SQLlite DB can be queried to retrieve deleted items ❗Undark. For Win10.
  2. LabCIF DOI - Detection of Objects in Images. YOLOv3 - You only look once. Uses Darknet in GPU mode (if CUDA available) or OpenCV in CPU mode for best performance. https://github.com/labcif/doi.
  3. Camera Fingerprint PRNU. To determine that the photo was taken by the suspect camera. Uses photo response non-uniformity of camera sensor: photo - noise = PRNU. Physical analysis of camera sensor + PRNU. Each sensor pixel is unique. Creates a description of camera sensor (fingerprint). To create a fingerprint, at least 30 photos are needed (white surface such as walls and clouds). Add evidence photos to data source. Select a folder with reference photos in ingest config. If the suspect photo was edited or resized, the information might be inaccurate (check tampering with other tools).
  4. SpeechToText. Type identifiaction module should be run before using this.
  5. BlackVue Geolocation.
  6. Android Usagestats. XML with user interatcion, moving apps to fore- and background and config changes. data/system/usagestats, protobuf files.
  7. TikTok for Android Analyzer. Contacts and Messages. tiktok.py. Move it to InternalPythonModules/android. Open module.py and import the tiptok.py and add the analyzer to the exisiting list.
  8. Forensic Analysis for Mobile Apps (FAMA). Extract user data from a connected Android device (using adb). Produce detailed reports (maps, timelines,multimedia). Recover rows from SQLite (Undark and SQL Deleted Records Parser). Https://github.com/labcif/FAMA. Clone repo, copy contents to autopsy/python_modules. Select live extraction with ADB.
  9. Notification analyser. Windows 10 Notification with “Your Phone” integration. %LOCALAPPDATA%\Microsoft\Windows\Notifications. SQLite3 DB produces WAL (Write Ahead Log) and SHM files. Notifications live for 3 days. Deleted rows can be recovered. YPA (anothe Autopsy plugin - https://github.com/labcif/ypa).
  10. Tinder for Android. Contacts and messages. tinder.py.
  11. Document Metadata.
  12. Nintendo Switch Forensics.
  13. iOS_sysdiagnose.
  14. Mass Export. Export of files of certain types into Case folder.
  15. Ring central.
  16. Wordlist. After keyword search.
  17. Google Takeout.
  18. AD1_Extractor.
  19. LevelDB.
