RSA

domain: crypto

doctype: article

To my utter embarrassment I had to admit that I’ve recently realized I didn’t remember how the RSA worked…. What I though to be RSA turned out to be Diffie-Hellman algorithm… Oops 😬. So, I’ve rewatched Khan’s video about RSA and was surprised I could every confuse the two. This article and another one about Diffie-Hellman is my way of explaining cryptographic notions: visual representation, verbal explanation and… code. Thanks to Python 🐍.

Diffie Hellman key exchange 🔑 is based on the following mathmetical laws and theorems:

  1. modulus arithmetics
  2. prime numbers
  3. raising to power

This algorithm is not encrypting anything! It’s used to generate the same key without sending it over network in order to use symmetric encryption algorithm. So, basically, it’s assymmetric algo for symmetric algo.

RSA is based on several mathematical … things:

  1. module exponantiation (as in Diffie Hellman’s algo)
  2. prime factorization and phi. Diffie Hellman uses factorization as well, but doesn’t use phi of numbers.
  3. raising to power (as in Diffie Hellman’s algo)
  4. Extended Euclidean algorithm (this one is also RSA personal stuff)

It is used to encrypt the message. Only assymetric algo is used, only hardcode hardcore!

Probably because these two guys share some basic principles which they are both relying on, I’ve been confusing them. When I was trying to untangle each one of these, I’ve found it very helpful to write down code while listening the explanation. Writing code requires full understanding of an algorithm in order to run it. When I thought I’ve got the main idea and started writing code, I’ve realised, that I was wrong…. It was still a long way to go. I also tried to explain it to my husband and it turned out to be even harder. And as the intro to this site states, you don’t understand a subject if you can’t explain it to a 6 year old. Well, I’ll be trying to explain it as easy as I can and since 6 year old are unlikely to read this article anyway, I’m going to raise the age of my target audience to at least 17-20 year olds (when people might start do some coding). To make it easier to understand and easier to code, I’m going to use very small numbers. I’ll be giving a small coding task along the way in order to keep the reader follow my chain of thoughs as well as to make sure it’s absolutely clear. A good starting analogy for this algorithm was given in the video on Khan academy: while other cryptography algorithms (symmetric ones) use a key both to lock and unlock the “message box”, this one uses a lock 🔒 to encrypt the message and a key 🔑 to decrypt. What does it mean? I’ll try to explain this using a what seems to be a very realistic example.

The Story of Evelyn

Let’s imagine a system adminstrator intern Evelyn 👨‍💻 who has leaked a DB of client passwords to a hacker 🐈 that resulted in a massive damage to the company 🤑 because the hacker managed to steal lots of client money that the company needs to compensate now. This company’s director 👨🏽‍💼 turned out to be a hard nut 🌰 and hired a forensic investigator 🕵️‍♀️ to help with this matter. The company’s director 👨🏽‍💼and the forensic investigator 🕵️‍♀️ are exchanging messages over corporate network which is of course sniffed by Evelyn 👨‍💻 . They are using RSA as their encryption method and we are going to test, whether he has a chance to decrypt the messages.

Locking the Message

For the forensic investigator 🕵️‍♀️, let’s call her Veronica (why not? 😆), everyone in the company is a suspect, including system administrator and his interns (Evelyn 👨‍💻 is, of course, one of them). That’s why she insisted that the company’s director 👨🏽‍💼, let’s call him Anduin (again, why not?), and herself encrypt the weekly status reports before sending. She proposes RSA since the messages are going to be small (only the name of the current suspect). Why size matters? Well, for this I’ve posted a separete small article here.

Anduin 👨🏽‍💼 and Veronica 🕵️‍♀️ don’t have any keys yet. They need to generate them and then may be exchange… (a little spoiler, yes and no). But how are they going to do so over the network which they both suspect to be sniffed by the person responsible for the data breach 💔? Keep in mind that Evelyn 👨‍💻 is a system administrator of the network and can see all messages sent over internal network. He will try to decrypt the messages that are exchanged by Veronica and Anduin or at least determine whether the message contains his name or not.

So, Veronica 🕵️‍♀️ want’s to let Anduin 👨🏽‍💼 that there are no results there yet. She writes down a message Nothing yet and converts it to a digit. Since any letter, digit or a symbol can be converted into a number (computers, binary and staff), this message looks like this in binary:

Binary data is hard for a human to operate with, so I’ll use hex (16 base integers):

def isPrime():
    if (n <= 1) : 
        return False
    if (n <= 3) : 
        return True
  
    # This is checked so that we can skip  
    # middle five numbers in below loop 
    if (n % 2 == 0 or n % 3 == 0) : 
        return False
    i = 5
    while(i * i <= n) : 
        if (n % i == 0 or n % (i + 2) == 0) : 
            return False
        i = i + 6
    return True

Creating the Key

The following stage is private for Anduin. He needs to craft a key 🔑 to unlock the lock 🔓 he has sent and read the message. So, it should be some operation that let’s him reverse the operation Veronica applied to the message using some information he knows but no one else knows. What information is that?

Did you wonder, why we needed to first choose two primes, multiply them and then share the result? And why both of them are marked as private in code? (in Python __ before an attribute means private, so when you instanciate the class you can’t get or set the value of these variables directly, only using some public function of this class).

Step 1. Alice 👧creates an open lock and the key. The lock will be sent to Bob 👨, and the key is private.

Trapdoor one-way function.

m ^ e mod N = c

c - cipher text, m - message. So, if m=0x6d and e is 3:

0x6d ^ 3 mod N = c

What’s the N? Alice 👧chooses two huge prime numbers of the same size (over 250 symbols) P1 = 3 and P2 = 5. Multiplies them and get’s N = 15. Now, she knows the prime factorisation of N which P1 and P2. Alice 👧shares N with Bob 👨. Eve’s got a copy, of course.

What’s the public exponent e? Alice 👧chooses this (usually, some small number) and shares with Bob 👨. Eve’s got a copy, of course.

0x6d ^ 3 mod 15 = c - everything in bold is the open lock 🔓 we talked about.

109 ^ 3 mod 15 = 4, which is 0x04 in hex. Now, Alice 👧has got 0x04. What is she supposed to do with it?

Remember P1 = 3 and P2 = 5? N = 15.

F(N) = F(P1-1) * F(P2-1) = F(3-1) * F(5-1) = 2 * 4 = 8

m ^ F(n) = 1 mod n

5^8 = 1 mod 15

They should not share a common factor (m and n). e should not share a common factor with F(n) and should be odd. m and N should not share a factor. So, 3 and 15 or 5 and 15 don’t work.

2 ^ 8 mod 15 = 1 always. But it’s not 390625 mod 15 = 10… .

1 raised to any power is 1

so m ^ k*F(n) = 1 mod n. So, 5 ^ 8 mod 15 = 1 is the same as 5 ^ k*8 mod 15 = 1

1 * any number = any number. So, m*m ^ kF(n) = m mod n is the same as m ^ kF(n) = 1 mod n. 5 ^ k8 = 1 mod 8 is the same as 5*5 ^ k8 = 5 mod 8

m * m ^ k * F(n) = m mod n which is the same as m ^ k * F(n) + 1 = m mod n. And with our digits: 5 ^ k*8 + 1 = 5 mod 15.

If e*d = k * F(n) + 1, then to find d we need to divide k*F(n) +1 by e.
