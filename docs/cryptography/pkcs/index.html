PKCS#5 padding

Created: 27.04.2023

This article describes how PKCS#5 padding works and how it can be exploited. This method is often used with block ciphers such as AES of DES.

How it works

With block ciphers the text needs to be of specific size that is is a multiple of a block length. Obviously, often it is not. How to make it work? Add something to the end of the message. But what? How would the recepient know when the message ends and the padding starts? Imagine we have the following end of the message padded with just 0s:

45 43 56 44 20 34 0 0 0 0

How does one know that these zeroes are not part of the message itself? Or maybe just some of these zeroes are? Consider now the following example.

45 43 56 44 20 34 4 4 4 4

So, the recepient sees repeated 4s. How do they know that this padding and not the actual contents? Easy. How many 4s? Four. Coincidence? No 😉 See another example below.

45 43 56 44 20 5 5 5 5 5

Now, the value of the padding is 5 and the number of 5s is … surprise-surprise… 5!

❓What if the message length is already a multiple of the block length? We have to add a full padding to it 🤷🏻‍♀️.

How is it exploited

With the case of AES (CBC) on needs a key and an IV to encrypt the message.

Say the attacker controls the IV, cipher text and can send the modified value to the receipent for validation for as long as they need.
