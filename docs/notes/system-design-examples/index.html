Designing Yelp!

Step 1. Requirements

Functional requirements:

  • Users can add reviews and like reviews of other users
  • User can upload photos along with their reviews
  • Business must have some like vs dislike functionality?
  • Can users add businesses?
  • Should the reviews be moderated and checked for falcification?
  • They data should be kept forever or at least until the business exists

Non-functional requirements:

  • Read-heavy
  • Some latency between uploading a photo/review and viewing it is acceptable
  • It should be highly reliable and users cannot loose their photo

Extended requirements:

  • Analytics
  • Private modes and security of information

Step 2. Estimations

description value
Users 1000 000
Reviews created per day 10 000
Reviews read per day 50 000
Average size of a review 0.1 Mb
Avergae size of an image 0.5 Mb
Photos per review 3
  1. DB size for 5 years. 10 000 * 365 * (0.5*3 + 0.1) = 1825 * 1.6 = 5840000 Mb = 5703 Gb = 5.5 Tb
  2. Read bandwidth. (0.5*3 + 0.1 = 1.6 Mb) the size of one review. 50 000 reviews read per day. 50 000 * 1.6 Mb = 80 000 Mb/day = 0.9259 Mb/sec
  3. Write bandwidth. 0.9259 Mb/sec / 5 (since the ration is 1/5) = 0.18518 Mb/sec.

Step 3. High-level design

I would have separate application servers for ordinary users and for businesses. As for images I would store some thumb nails for immediate rendering and if the user wants to see the full picture with high resolution, query my S3 server.

Step 4. Database Schema

I would have one DB for users, one DB for businesses and a separate DB for media special efficiency algorithm, I would use an Amazon S3 object DB for that. HDFS is probably an overkill â“ DBs for Users and Businesses would be SQL since there are lots of interconnections. And I also would have a user preference DB (NoSQL for efficient working with BigData).
